RAP project (Non-Skippable Adverts For Whole Swiss TV Industry)New ProjectAWS Cloud

Overview: AWS system for introducing non-skippable adverts in to the whole Swiss TV industry for on-demand TV. It came about due to a change in broadcasting legislation because consumers were previously able to skip ads which loses ad revenue.

RWEL Ltd was contracted to create the shared APIs & backend used throughout the industry and by consumer devices. An ad sales house will book a campaign and the system ultimately tries to ensure that the target impressions for the ad campaign is delivered. The system specification was quite vague and needed a lot of discussions, especially because there are so many clients. RWEL Ltd did all of the AWS development.

The APIs handle 22m requests per day.

Used by: Whole Swiss TV industry (operators, broadcasters, ad sales, audience measurement) & any consumer device capable of playing on-demand TV (STB, browser, mobile application). Companies will use the APIs to upload their data (Ad campaigns, channels, available impressions, media, EPGs etc) or try to book a campaign (if sufficient impressions available) or read data (E.g., Campaign progress or reports etc). Consumer devices will use the API to request an ad list to play out and to send playout progress events so that the system can count an impression/contact.

Components: REST & SOAP APIs (ASP.NET Core Web API) for consumer devices (Request ads, notify playout events etc), APIs for managing data (Create campaigns, managing static data, manage impression inventory, APIs for reporting, APIs for monitoring information (E.g., Health checks), monitoring background tasks (E.g., Send emails, create Jiras for issues, send Datadog metrics, send MS Teams notifications), general background tasks (E.g., Campaign managment & planning, data caching, import EPGs, archiving etc), RavenDB database, Redis database & Confluence documents (Technical information, "How To" guides, troubleshooting guides etc)

My role: I led the backend development (Team of 3). I did a lot of design work. I was responsible for getting the clients to agree on the detailed design for many of the component components due to the system specification being high level & vague. It was a very small team and I also did a lot of development work.

EPoS OMR project (Implement OMR Betting Slips For Shops)New Project

Overview: Project to implement OMR betting slips for shop systems. E.g., Horse racing special events like Grand National, lottery & football coupons. It enables bets to be automatically added to the system without the shop staff having to translate what was written on the betting slip. Manually translating bets is a very slow process otherwise.

The customer will fill in a pre-printed betting slip (E.g., Select 5 lottery numbers), the shop staff will feed it through the document scanner, a DLL will identify which type of betting slip from the barcode and then return a list of which items were ticked on the slip. The printed receipt will list which items were ticked and display the cost of the bets.

Used by: Ladbrokes shops & independent bookmakers shops.

Components: DLL (C++) for recognising pre-printed betting slips (Identifies which marks were made and produces XML), a test harness for the DLL and a UI for designing OMR betting slips in Visio format.

My role: I did about 40% of the work on the project and worked on all components. I did a lot of work on improving the accuracy of the DLL.
